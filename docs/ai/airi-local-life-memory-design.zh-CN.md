# AIRI 本地生活记忆系统设计文档

> 文档角色：背景附录 / 产品设计材料。
> 当前主计划见 [`airi-memory-agent-orchestrator-design.zh-CN.md`](./airi-memory-agent-orchestrator-design.zh-CN.md)。
> 后续工程路线、优先级和实现状态以主计划为准。

> 目标：让 AIRI 不只是一个会聊天的桌面角色，而是一个能在本地记录用户生活、行为习惯、项目经历和共同回忆的长期陪伴型智能体。

## 1. 核心判断

这个项目非常适合做“本地生活记忆系统”。原因是 AIRI 的定位天然不是一次性问答工具，而是长期存在于桌面的陪伴对象。长期陪伴的关键不是每次都说漂亮话，而是它能逐渐理解用户：

- 用户是谁，现在处于什么阶段。
- 用户最近在做什么项目。
- 用户习惯什么时候学习、什么时候疲惫。
- 用户喜欢怎样的提醒方式。
- 用户有哪些明确边界，例如哪些文件夹不能碰。
- 用户曾经和 AIRI 共同经历过哪些重要时刻。

这些内容不应该全部写进大模型参数里，而应该存储在用户本地的记忆系统中。大模型负责理解和表达，记忆系统负责保存和检索，Agent 编排层负责决定何时使用记忆。

## 2. 产品定位

本地生活记忆系统可以让 AIRI 成为三种角色的结合体：

| 角色 | 能力 |
| --- | --- |
| 陪伴者 | 记得用户的状态、偏好和经历，能自然延续过往话题 |
| 桌面助理 | 记得项目、服务、路径、常用工具和工作流 |
| 成长记录者 | 帮用户整理学习、转码、项目推进和生活片段 |

最终体验应该接近：

> 用户不是每次都从零开始解释自己是谁，AIRI 能带着过去的共同经历继续对话。

## 3. 为什么必须本地存储

生活记忆比普通聊天记录更敏感。它可能包含：

- 学习和职业规划。
- 情绪状态。
- 日常作息。
- 文件路径。
- 项目秘密。
- 人际关系。
- 语音转文字记录。
- 桌面操作历史。

因此第一原则是本地优先：

- 默认不上传云端。
- 默认不用于外部训练。
- 用户可以查看、修改和删除所有记忆。
- 切换云端 LLM 时，必须明确提示会发送哪些上下文。
- 高敏感记忆需要用户确认后保存。

## 4. 记忆分层

建议将 AIRI 的记忆分为六层。

### 4.1 核心身份记忆

这类记忆描述用户长期稳定的背景。

示例：

- 用户本科为土木工程背景。
- 用户目前是复旦大学智能机器人与先进制造创新学院学生。
- 用户正在转码，目标方向偏 AI / 大模型 / 智能体 / 算法岗。
- 用户维护 AIRI + Gemma 本地化项目。

特点：

- 重要性高。
- 更新频率低。
- 修改时最好由用户确认。

### 4.2 偏好与边界记忆

这类记忆决定 AIRI 如何与用户相处。

示例：

- 用户希望回答清楚、易懂，不要堆术语。
- 用户希望项目能服务转码和面试。
- 不要碰 MiniBox 项目文件。
- 不要碰毕业设计文件夹。
- 涉及删除文件、结束进程、执行命令时必须先确认。

特点：

- 优先级最高。
- 应在每次 Agent 决策前注入。
- 不允许被普通对话轻易覆盖。

### 4.3 行为习惯记忆

这类记忆描述用户的长期行为模式。

示例：

- 用户经常在晚上推进 AIRI 项目。
- 用户喜欢先看总体路线，再进入代码。
- 用户对大模型微调、记忆系统、Computer Use 很感兴趣。
- 用户更关注项目能否变成面试作品，而不是只做玩具 demo。

特点：

- 从多次对话和操作中总结。
- 需要置信度字段。
- 错误总结应允许用户纠正。

### 4.4 项目状态记忆

这类记忆记录 AIRI 项目的技术状态。

示例：

- Ollama 运行在 11434。
- GPT-SoVITS 运行在 9881。
- Whisper / FunASR STT 服务运行在 8000，二选一。
- GSV 根路径返回 404 也可以视为在线。
- GSV 的 `streaming_mode` 需要设为 `false`。

特点：

- 和项目开发强相关。
- 可从交接文档、代码修改、服务检查中生成。
- 可以用于桌面助理和故障排查。

### 4.5 事件与回忆记忆

这类记忆记录“发生过什么”。

示例：

- 某天完成了 AIRI + Gemma 微调技术报告。
- 某次修复了 GSV 无声音问题。
- 用户提出想把 AIRI 做成能记录生活习惯和回忆的陪伴系统。
- 用户确认自己的真实画像需要从“土木学生”更新为“FDU 智能机器人与先进制造创新学院学生，正在转码”。

特点：

- 可以按时间线查看。
- 适合生成“我们最近做了什么”的回顾。
- 可以成为微调数据和项目展示材料来源。

### 4.6 短期工作记忆

这类记忆只在当前任务或短时间内有效。

示例：

- 当前正在整理 `.gitignore`。
- 当前正在设计本地生活记忆系统。
- 当前还没有运行 typecheck。

特点：

- 生命周期短。
- 可在任务结束后摘要进长期记忆。
- 不需要永久保存全部细节。

## 5. 数据结构设计

第一版推荐使用 PGlite + Drizzle，运行在 Electron main process，数据保存在 `app.getPath('userData')/memory/pglite`。它适合本地 Electron 桌面应用，能避免 Windows native SQLite/better-sqlite3 打包风险，也方便后续迁移、索引和可选向量检索。

### 5.1 记忆表

```sql
CREATE TABLE memory_items (
  id TEXT PRIMARY KEY,
  type TEXT NOT NULL,
  scope TEXT NOT NULL,
  content TEXT NOT NULL,
  summary TEXT,
  importance INTEGER NOT NULL,
  confidence REAL NOT NULL,
  source TEXT NOT NULL,
  status TEXT NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  last_used_at TEXT,
  expires_at TEXT,
  metadata_json TEXT
);
```

字段说明：

| 字段 | 含义 |
| --- | --- |
| `type` | identity、preference、boundary、habit、project、event、working |
| `scope` | life、project、system、conversation |
| `importance` | 1-5，越高越优先 |
| `confidence` | 0-1，表示模型总结是否可靠 |
| `source` | user_confirmed、conversation、system、imported |
| `status` | active、archived、rejected、needs_review |
| `expires_at` | 可选，短期记忆过期时间 |

第一版应预留以下字段或等价 metadata，用于后续 LLMWiki/RAG/云端隐私控制：

| 字段 | 含义 |
| --- | --- |
| `privacy` | public、local、sensitive、secret |
| `cloud_allowed` | 是否允许发送给云端模型 |
| `rag_index` | 是否进入 RAG 检索 |
| `wiki_page_path` | 关联的 LLMWiki Markdown 路径 |
| `source_memory_ids` | 合并/摘要来源记忆 |
| `last_exported_at` | 最近一次导出到 LLMWiki 的时间 |

### 5.2 会话表

```sql
CREATE TABLE conversation_messages (
  id TEXT PRIMARY KEY,
  conversation_id TEXT NOT NULL,
  role TEXT NOT NULL,
  content TEXT NOT NULL,
  created_at TEXT NOT NULL,
  metadata_json TEXT
);
```

### 5.3 事件时间线表

```sql
CREATE TABLE life_events (
  id TEXT PRIMARY KEY,
  title TEXT NOT NULL,
  description TEXT NOT NULL,
  happened_at TEXT NOT NULL,
  importance INTEGER NOT NULL,
  related_memory_ids_json TEXT,
  created_at TEXT NOT NULL,
  metadata_json TEXT
);
```

### 5.4 记忆审计表

```sql
CREATE TABLE memory_audit_logs (
  id TEXT PRIMARY KEY,
  memory_id TEXT,
  action TEXT NOT NULL,
  old_value_json TEXT,
  new_value_json TEXT,
  created_at TEXT NOT NULL
);
```

这张表用于记录记忆是如何被创建、修改、删除和恢复的。对于“长期陪伴”来说，用户必须能追溯 AIRI 为什么记住了某件事。

## 6. 记忆写入流程

不应该把每句话都永久保存为重要记忆。推荐流程如下：

```text
对话结束或任务结束
  -> 生成会话摘要
  -> 提取候选记忆
  -> 判断记忆类型和重要性
  -> 低风险记忆自动保存
  -> 高敏感/高优先级记忆进入待确认
  -> 用户可在记忆管理页查看、编辑、删除
```

需要确认的记忆包括：

- 用户身份变化。
- 职业目标。
- 明确禁止操作的路径。
- 隐私信息。
- 情绪或健康相关长期判断。
- 可能影响 Agent 行为的规则。

可以自动保存的记忆包括：

- 项目进展摘要。
- 用户明确表达的普通偏好。
- 当前任务状态。
- 非敏感技术配置。

### 6.1 从聊天记录和本地知识库导入

AIRI 可以通过用户手动导入的微信聊天记录、飞书聊天记录和本地知识库来建立更贴近用户的长期陪伴能力。这里的目标不是“复制聊天记录”，而是提取可审阅、可删除、可追溯的候选记忆。

推荐导入流程：

```text
用户选择导入源
  -> 保存到本地 raw import 区
  -> 解析格式
  -> 脱敏第三方信息
  -> 按会话、时间、主题切块
  -> 生成摘要
  -> 提取候选记忆
  -> 用户审核
  -> 写入 Memory Service
  -> 需要时导出到 LLMWiki / Obsidian
```

导入源职责：

| 导入源 | 主要价值 | 默认处理 |
| --- | --- | --- |
| 微信聊天记录 | 生活回忆、沟通风格、关系上下文 | 高隐私，默认进入审核队列 |
| 飞书聊天记录 | 项目状态、协作习惯、任务历史 | 按项目记忆处理，避免自动人格化 |
| 本地知识库 | 论文、笔记、项目资料、面试资料 | 可进入 RAG，重要摘要再进 LLMWiki |

人格塑造需要分层处理：

- 用户画像：背景、目标、长期偏好，必须可编辑。
- 沟通风格：喜欢详细解释、类比、步骤化、还是简短回答，建议通过多次样本统计后确认。
- 共同回忆：重要事件和项目节点，适合进入时间线。
- 知识背景：本地文档和项目资料，主要用于 RAG，不等同于人格。
- LoRA 候选：只保存脱敏后的高质量问答、记忆使用样本和工具调用样本。

隐私边界：

- 原始聊天记录不进入 Git。
- 原始聊天记录默认不发给云端模型。
- 涉及第三方个人信息的内容默认标记为 `sensitive`。
- 情绪、关系、性格判断只能作为候选记忆，必须由用户确认。
- 用户删除导入批次时，应能删除该批次产生的候选记忆和派生摘要。

## 7. 记忆检索流程

对话前，系统应该检索少量相关记忆，而不是把所有内容都塞进 prompt。

推荐流程：

```text
用户输入
  -> 判断意图：聊天、项目协助、桌面操作、回忆查询
  -> 检索强规则记忆
  -> 检索相关生活/项目记忆
  -> 检索最近会话摘要
  -> 排序并压缩上下文
  -> 注入 LLM
```

排序因素：

- 相关性。
- 重要性。
- 新鲜度。
- 置信度。
- 是否用户确认。
- 是否属于强边界规则。

## 8. 生活回忆能力

这是这个 idea 最有温度的地方。AIRI 可以提供“共同回忆”能力，例如：

- “我们这个月做了哪些事情？”
- “我最近是不是一直在焦虑转码？”
- “你还记得我为什么想微调 Gemma 吗？”
- “帮我总结一下这段时间 AIRI 项目的进展。”
- “我上次说想把你做成什么样？”

这类能力可以由 `life_events`、`memory_items` 和会话摘要共同支持。

输出形式可以包括：

- 时间线。
- 月度回顾。
- 项目周报。
- 情绪和学习状态总结。
- “我们共同完成的事情”。

## 9. 行为习惯分析

行为习惯不能靠一次对话就下结论，应通过多次观察逐渐形成。

示例规则：

```text
如果某类行为在 7 天内出现 3 次以上，生成候选 habit memory。
如果用户纠正过该习惯，降低置信度或归档。
如果该习惯会影响提醒方式，需要用户确认。
```

可以分析的习惯：

- 常用工作时间。
- 常见任务类型。
- 喜欢的解释风格。
- 容易卡住的环节。
- 常用工具和服务。
- 对陪伴语气的偏好。

必须避免的错误：

- 不要根据少量样本给用户贴标签。
- 不要把短期情绪当成长期性格。
- 不要保存过度敏感的生活判断。
- 不要用“你总是”“你一定”这种绝对化表达。

## 10. 隐私与用户控制

本地生活记忆系统必须有一个清晰的控制界面。

第一版 UI 至少应支持：

- 查看全部记忆。
- 按类型筛选记忆。
- 搜索记忆。
- 编辑记忆。
- 删除记忆。
- 标记“不要再使用这条记忆”。
- 导出记忆。
- 清空记忆。
- 查看记忆来源。

后续前端体验建议升级为 **Obsidian-inspired Memory Workspace**，而不是只停留在传统设置页：

- 左侧：记忆库、导入来源、Review Inbox、项目、人物、标签和导出包。
- 中间：Markdown 记忆正文、候选记忆、共同回忆时间线或导入批次详情。
- 右侧：来源证据、置信度、隐私等级、反链、相关记忆、RAG 使用状态、LoRA/梦境候选状态。
- Obsidian/AIRI-Brain vault 作为人类可读层，支持用本地 Obsidian 打开、手动整理和 Graph View 浏览。
- AIRI 内置管理台仍然负责安全动作：确认、拒绝、合并、删除、脱敏、公开画像导出和训练候选导出。

这意味着 Obsidian 是优秀的外部前端和体验参考，但不是 AIRI 的必需运行依赖。Memory DB 仍然是事实源，Markdown/Obsidian 只作为可读视图、人工整理层和受控回流入口。

建议增加三个开关：

| 开关 | 含义 |
| --- | --- |
| 自动记录会话摘要 | 是否在对话后生成摘要 |
| 自动提取长期记忆 | 是否自动保存低风险记忆 |
| 敏感记忆必须确认 | 高敏感内容是否总是进入待确认 |

云端模型调用时，应该显示提示：

> 本次请求会把以下记忆片段发送给云端模型。是否继续？

## 11. 与 LoRA 微调的关系

本地生活记忆系统可以反过来帮助 LoRA 微调。

它可以沉淀四类训练数据：

- 陪伴聊天优质样本。
- 记忆检索后回答样本。
- 用户纠正 AIRI 的偏好样本。
- 工具调用和安全确认样本。

训练目标不是让 Gemma 背下用户隐私，而是让它学会：

- 如何自然使用检索到的记忆。
- 如何根据用户习惯调整表达。
- 如何在不确定时询问。
- 如何拒绝或确认高风险操作。
- 如何保持 AIRI 的人格一致性。

因此，长期记忆系统是 LoRA 数据飞轮的前置条件。

## 11.5 LLMWiki 与 Obsidian 的关系

Memory Service 是事实源，LLMWiki/Obsidian Markdown 是可读导出层。

也就是说：

- 数据库保存结构化记忆、状态、来源、置信度、审计记录。
- LLMWiki 把稳定、用户确认、适合长期使用的记忆整理成人类可读 Markdown。
- Obsidian 可以直接打开 AIRI-Brain vault，用来查看、编辑、整理知识。
- 手动编辑 Markdown 不能静默覆盖数据库，只能作为 `manual_override` 或 `import_candidate` 回流到 Memory Service。
- `secret` 级别记忆不导出、不进入 RAG、不发送给模型。

## 12. 与 Computer Use 的关系

Computer Use 会产生大量行为记录，但不是所有记录都应该变成记忆。

可保存：

- 用户常用应用。
- 用户常用项目路径。
- 反复出现的工作流。
- 需要长期遵守的操作边界。

不默认保存：

- 密码、token、密钥。
- 私密网页内容。
- 具体聊天对象内容。
- 未经授权的文件正文。
- 一次性的临时窗口信息。

Computer Use 的历史记录应该分为两类：

- 审计日志：用于安全追溯，默认不进入 LLM 上下文。
- 生活/项目记忆：经过筛选和摘要后才进入记忆系统。

## 13. 第一阶段实现范围

建议第一阶段不要做太大，只做最小可用闭环。

### 必做

- PGlite + Drizzle 本地数据库。
- 记忆 CRUD。
- 会话摘要表。
- 记忆检索函数。
- 对话前注入相关记忆。
- 设置页中的记忆查看/删除。

### 暂不做

- 复杂向量数据库。
- 自动控制所有桌面应用。
- 自动生成可执行代码。
- 自动训练 LoRA。
- 多用户同步。

第一阶段目标：

> AIRI 能记住用户确认过的核心信息，并能在后续对话中自然引用。

## 14. 后续阶段

### P1：可控记忆

- 手动添加记忆。
- 自动保存低风险项目记忆。
- 敏感记忆待确认。

### P2：回忆时间线

- 按日期展示事件。
- 生成周报/月报。
- 支持“我们最近做了什么”。

### P3：习惯画像

- 从多次会话中提取习惯。
- 支持置信度和用户纠正。
- 用习惯调整提醒方式。

### P4：Agent 联动

- 让记忆参与工具调用决策。
- 操作电脑前检查边界规则。
- 根据项目记忆辅助故障排查。

### P4.5：多数据源导入

- 支持用户手动导入微信聊天记录、飞书聊天记录和本地知识库。
- 原始导入数据只保存在本地 raw import 区。
- 对内容进行脱敏、切块、摘要和候选记忆提取。
- 用户确认后再写入长期记忆、时间线、LLMWiki 或 RAG。
- 删除导入批次时，同步清理派生候选记忆和摘要。

### P5：LoRA 数据飞轮

- 从真实对话中筛选训练样本。
- 从已审核的导入内容中提取沟通风格和记忆使用样本。
- 构造记忆使用样本。
- 微调 Gemma，让它更会使用记忆和陪伴用户。

## 15. 面试表达

这个 idea 在面试中可以这样讲：

> 我没有把长期陪伴简单理解成“给模型写一个人设 prompt”。我把它拆成了三层：模型微调用来学习表达风格和工具调用习惯，长期记忆系统用来保存动态用户信息，Agent 编排层负责在合适的时机检索和注入记忆。这样既保护隐私，又能让端侧小模型具备长期陪伴能力。

如果面试官问“为什么不用微调来记住用户”，可以回答：

> 用户记忆是动态、可删除、可审计的数据，不适合固化进模型参数。微调适合学习稳定行为模式，记忆适合保存不断变化的事实。把两者分开，系统才可控。

如果面试官问“如何避免隐私问题”，可以回答：

> 记忆默认本地存储，用户可以查看和删除；敏感记忆需要确认；云端模型调用前会显示将发送的记忆片段；Computer Use 记录先进入审计日志，只有经过筛选和摘要的内容才会成为长期记忆。

## 16. 总结

这个本地生活记忆系统是 AIRI 项目的核心进化方向。它让 AIRI 从“每次重新认识用户的聊天模型”变成“能和用户共同积累经历的本地陪伴智能体”。

最重要的设计原则是：

- 记忆本地化。
- 用户可控制。
- 敏感内容需确认。
- 微调学习行为，记忆保存事实。
- Computer Use 记录必须经过安全筛选。

沿着这条路线继续做，AIRI 不只是一个桌宠项目，而会成为一个完整的端侧个人智能体项目，既有产品温度，也有算法和系统工程深度。
