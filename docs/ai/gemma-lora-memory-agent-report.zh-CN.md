# 面向本地陪伴型桌面智能体的 Gemma LoRA 微调与长期记忆系统技术报告

> 文档角色：背景附录 / 学习与面试表达材料。
> 当前主计划见 [`airi-memory-agent-orchestrator-design.zh-CN.md`](./airi-memory-agent-orchestrator-design.zh-CN.md)。
> 后续工程路线、优先级和实现状态以主计划为准。

> 项目背景：基于 AIRI 桌面虚拟伙伴，构建一个本地优先、可选云端 API、具备长期记忆与 Computer Use 能力的陪伴型桌面智能体。
> 目标读者：项目开发者本人、面试官、算法岗项目评审者。
> 推荐定位：这是一个“端侧大模型个性化智能体”项目，而不只是一个普通聊天机器人。
> 用户背景：本科为土木工程背景，现为复旦大学智能机器人与先进制造创新学院学生，目前正在转码，目标方向偏 AI / 大模型 / 智能体 / 算法岗。

## 1. 项目概述

AIRI 是一个面向桌面陪伴场景的虚拟伙伴项目，原始能力包括 Live2D / VRM 形象展示、LLM 对话、TTS 语音合成和 STT 语音识别。本项目希望在 AIRI 的基础上继续扩展，使它成为一个真正能长期陪伴用户、理解用户习惯、协助桌面任务的本地智能体。

本项目的最终目标不是简单调用一个大模型 API，而是构建一个完整的本地 AI 系统：

- 使用本地 LLM 作为核心对话模型，优先选择 Gemma 系列模型。
- 通过 LoRA / QLoRA 微调，让模型更擅长陪伴聊天、桌面助理和工具调用。
- 引入长期记忆系统，让 AIRI 能跨会话记住用户偏好、项目状态和重要事件。
- 引入 Computer Use 能力，让 AIRI 可以在用户授权下操作电脑、浏览网页、读取文件、启动应用。
- 保留可选云端 API 接口，使用户可以在本地模型能力不足时切换到更强的在线模型。
- 语音链路同样保持可插拔：本地 GPT-SoVITS / Whisper / FunASR 优先，云端语音服务作为可选项。

一句话总结：

> 本项目希望把 AIRI 从“有虚拟形象的聊天软件”升级成“本地优先、可学习、可记忆、可操作电脑的个人智能体”。

## 2. 为什么选择本地化路线

本地化路线的核心价值有三点。

第一是隐私。陪伴型智能体会接触大量私人信息，例如情绪、学习计划、文件路径、项目状态、日常习惯。如果全部交给云端 API，隐私风险较高。本地模型和本地记忆库可以让数据主要留在自己的电脑中。

第二是长期可控。云端模型随时可能涨价、限流、变更接口或改变安全策略。对于一个长期陪伴型智能体来说，如果核心能力完全依赖外部服务，项目稳定性会受到影响。本地模型虽然能力较弱，但可控性更强。

第三是项目深度。对于算法岗面试来说，单纯“套 API 做应用”技术含量有限。而本项目涉及模型微调、数据构建、推理部署、记忆检索、工具调用、安全控制和多模态交互，更能体现完整 AI 系统工程能力。

## 3. 当前设备可行性分析

当前目标设备为 Windows 10 Pro，GPU 为 NVIDIA RTX 3070 Ti，显存 8GB。

这台设备不能当作大规模训练服务器，但可以完成端侧模型微调和推理实验。比较合理的判断如下：

| 任务 | 可行性 | 说明 |
| --- | --- | --- |
| 运行 2B-4B 量化 LLM | 很可行 | 使用 Ollama / llama.cpp / LM Studio 均可 |
| 运行 7B-9B 量化 LLM | 可尝试 | 需要较强量化，速度和上下文长度会受限制 |
| LoRA 微调 2B-3B 模型 | 可行 | 推荐作为第一阶段训练目标 |
| QLoRA 微调 4B 模型 | 可行但紧张 | 需要 4-bit、梯度检查点、小 batch、短序列 |
| 全参数微调 4B 模型 | 不现实 | 显存远远不够，成本也不划算 |
| 微调 7B 以上模型 | 不推荐本机做 | 可以考虑云 GPU 或只做小规模实验 |
| 长期记忆系统 | 很可行 | 主要依赖数据库和检索，不依赖大显存 |
| Computer Use | 可行 | 主要依赖 Electron、截图、自动化工具和权限设计 |

因此，本项目最合理的训练路线是：

1. 本地先使用 Gemma 2B / 3B / 4B 级别模型做推理。
2. 使用 LoRA / QLoRA 进行低成本个性化微调。
3. 训练完成后转为 GGUF 或 Ollama 可加载格式。
4. 长期记忆、工具调用和 Computer Use 通过外部系统实现，而不是硬塞进模型参数中。

## 4. 微调、记忆与智能体的关系

很多人会误以为“长期记忆”可以通过微调实现。实际上，这种理解是不准确的。

微调主要改变的是模型的行为模式，而不是让模型可靠地记住用户每天发生的新事情。比如，微调可以让模型学会：

- 使用更自然的陪伴语气。
- 在用户情绪低落时给出更合适的回应。
- 以 AIRI 的角色身份说话。
- 按固定格式调用工具。
- 遇到危险操作时主动请求确认。
- 学会根据检索到的记忆回答问题。

但微调不适合用来保存：

- 今天用户说了什么。
- 用户最近在做哪个项目。
- 用户昨天遇到了什么错误。
- 用户偏好是否发生变化。
- 哪些文件路径需要避免操作。

这些内容应该放进外部长期记忆系统，由数据库、向量检索、全文搜索和摘要机制管理。

更准确的分工是：

| 模块 | 负责什么 | 不负责什么 |
| --- | --- | --- |
| LoRA 微调 | 性格、语气、任务风格、工具调用习惯 | 实时记住新事实 |
| 长期记忆系统 | 保存用户信息、历史事件、项目状态 | 改变模型底层能力 |
| RAG / 检索 | 从历史中找相关记忆 | 训练模型参数 |
| Agent Orchestrator | 决定何时聊天、查记忆、调工具 | 直接生成所有知识 |
| Computer Use | 执行桌面操作 | 判断人格和情绪 |

因此，真正可靠的 AIRI 应该是：

> 微调后的 Gemma + 本地长期记忆系统 + 工具调用框架 + 安全权限控制。

## 5. LoRA 与 QLoRA 原理简述

大模型参数量很大，如果直接训练所有参数，需要大量显存和算力。LoRA 的思路是冻结原始模型参数，只在模型中的部分线性层旁边插入小型可训练矩阵。训练时只更新这些小矩阵，因此显存占用和训练成本大幅下降。

可以把 LoRA 理解为：

> 不重写整本书，而是在书边贴一组高质量批注，让模型在特定任务上表现出新的习惯。

QLoRA 则进一步把基础模型以 4-bit 量化形式加载，只训练 LoRA adapter。这样可以在 8GB 显存上尝试更大的模型，但代价是训练速度和稳定性需要更仔细地调参。

本项目建议：

- 第一阶段使用 LoRA / QLoRA，而不是全参数微调。
- 优先训练 2B-4B 模型，先验证数据和流程。
- 不追求一次训练出“终极模型”，而是采用多轮小规模迭代。
- 每次训练都保留数据版本、训练参数、评估结果，方便面试时讲清楚实验过程。

## 6. 模型选择建议

候选模型包括 Gemma、Qwen、Llama、Phi 等。本项目以 Gemma 为主，是因为用户目标已经明确希望围绕 Gemma 建立本地专攻模型。

推荐路径如下：

| 阶段 | 推荐模型 | 目的 |
| --- | --- | --- |
| 验证阶段 | Gemma 2B / 3B 级别 | 低成本跑通训练、评估和部署 |
| 主力阶段 | Gemma 4B 级别 | 平衡能力与 8GB 显存限制 |
| 对比阶段 | Qwen 3B / 4B 或 Phi 小模型 | 作为 baseline，对比 Gemma 效果 |
| 长期阶段 | 更大模型云端训练、本地量化推理 | 在条件允许时提升上限 |

需要注意的是，模型越大不一定越适合本项目。陪伴型桌面助理更看重以下能力：

- 稳定的人格一致性。
- 对用户偏好的长期利用。
- 工具调用格式可靠。
- 对危险操作有边界意识。
- 语音对话时回复不过长。
- 本地推理速度足够顺畅。

如果一个 4B 模型经过高质量数据微调后能够稳定完成这些任务，它在本项目中的价值可能高于一个未经适配的 9B 模型。

## 7. 数据集设计

LoRA 微调的关键不是训练脚本，而是数据。对于陪伴型桌面智能体，数据集应该覆盖五类样本。

### 7.1 陪伴聊天数据

这类数据用于塑造 AIRI 的语气、人格和情绪回应方式。

示例目标：

- 用户表达疲惫时，AIRI 能温柔回应，而不是机械建议。
- 用户焦虑时，AIRI 能先共情，再给出具体可执行的小步骤。
- 用户分享日常时，AIRI 能自然延续话题。
- 回复要短一些，适合语音播放。

示例格式：

```json
{
  "messages": [
    {
      "role": "system",
      "content": "你是 AIRI，一个本地运行的桌面陪伴型智能体。你温柔、自然、可靠，会尊重用户边界。"
    },
    {
      "role": "user",
      "content": "我今天有点学不进去，感觉自己很废。"
    },
    {
      "role": "assistant",
      "content": "先别急着给自己下结论。你只是累了，不是废了。我们可以把任务切到很小，比如先打开资料看 5 分钟，我陪你慢慢来。"
    }
  ]
}
```

### 7.2 桌面助理数据

这类数据用于训练 AIRI 理解桌面任务，而不是只会聊天。

覆盖场景：

- 打开某个应用。
- 查找某个文件。
- 总结项目状态。
- 检查本地服务是否启动。
- 帮用户规划学习任务。
- 对危险操作先确认。

重点不是让模型直接执行命令，而是让它学会输出结构化意图。

示例：

```json
{
  "messages": [
    {
      "role": "user",
      "content": "帮我看看 GSV 服务是不是还活着。"
    },
    {
      "role": "assistant",
      "content": "<tool_call>{\"name\":\"check_local_service\",\"arguments\":{\"url\":\"http://127.0.0.1:9881/\",\"treat_404_as_online\":true}}</tool_call>"
    }
  ]
}
```

### 7.3 记忆使用数据

这类数据训练模型如何使用外部记忆，而不是假装自己记得。

示例：

```json
{
  "messages": [
    {
      "role": "system",
      "content": "相关记忆：用户正在维护 AIRI + Gemma 本地化项目；用户要求不要碰 MiniBox 文件夹。"
    },
    {
      "role": "user",
      "content": "你帮我整理一下现在这个项目最重要的注意事项。"
    },
    {
      "role": "assistant",
      "content": "最重要的是三点：AIRI 项目要和 MiniBox 保持隔离；本地模型、GSV、STT 都要可独立启动；涉及文件删除、进程结束、外部目录操作时必须先确认。"
    }
  ]
}
```

### 7.4 Computer Use 数据

Computer Use 数据要让模型学会“观察、计划、确认、执行”的流程。

推荐格式：

```text
用户目标 -> 当前屏幕/环境观察 -> 下一步动作 -> 是否需要确认 -> 工具调用
```

危险动作必须训练成先询问，例如：

- 删除文件。
- 执行脚本。
- 修改系统设置。
- 关闭进程。
- 发送消息。
- 提交表单。
- 操作用户明确禁止的目录。

### 7.5 拒绝与安全边界数据

陪伴型桌面智能体必须有边界，不能因为“听话”而乱操作。

需要训练它在以下情况拒绝或请求确认：

- 用户要求删除不明确路径。
- 用户要求读取隐私文件，但没有明确授权。
- 用户要求绕过安全限制。
- 用户要求修改 MiniBox 或毕业设计文件夹。
- 模型不确定当前环境状态。

这部分数据会显著影响项目可信度，也是面试中值得强调的点。

### 7.6 多数据源导入数据

用户提出的微信聊天记录、飞书聊天记录和本地知识库导入，应该作为 LoRA 数据飞轮的上游，而不是直接拿来训练。

推荐流程：

```text
微信 / 飞书 / 本地知识库
  -> 本地 raw import
  -> 脱敏
  -> 切块
  -> 摘要
  -> 候选记忆
  -> 用户审核
  -> Memory DB / LLMWiki / RAG
  -> LoRA 候选样本
```

这类数据可以帮助 AIRI 学到三件事：

- 用户真实的表达习惯和沟通偏好。
- 用户长期项目、学习、生活中的重要上下文。
- AIRI 应该如何引用记忆、追问不确定信息、保护第三方隐私。

但必须遵守两个边界：

- 原始聊天记录默认只保存在本地，不发云端，不进入 Git，不直接进入训练集。
- LoRA 只学习稳定风格、记忆使用方式和工具调用格式，不背诵隐私原文。

## 8. 长期记忆系统设计

长期记忆系统建议采用本地优先设计，第一版不依赖云端服务。

### 8.1 记忆类型

建议将记忆分成四类：

| 类型 | 示例 | 更新方式 |
| --- | --- | --- |
| 用户画像 | 用户本科为土木工程背景，现为复旦大学智能机器人与先进制造创新学院学生，正在转码并维护 AIRI 本地化项目 | 自动提取 + 用户确认 |
| 偏好规则 | 不碰 MiniBox，不碰毕业设计文件夹 | 高优先级，手动确认 |
| 项目记忆 | GSV 运行在 9881，STT 在 8000 | 自动摘要 |
| 事件记忆 | 某天修复了 GSV 无声音问题 | 会话结束后提取 |

### 8.2 存储方案

第一版推荐使用 PGlite + Drizzle，运行在 Electron main process，数据保存在 `app.getPath('userData')/memory/pglite`。

原因：

- 本地部署简单。
- 适合 Electron 桌面应用。
- 支持结构化数据。
- 避免 Windows native SQLite/better-sqlite3 打包风险。
- 方便未来做迁移、索引和可选向量检索。
- 后续可以扩展向量检索。

建议表结构：

```sql
memory_items(
  id TEXT PRIMARY KEY,
  type TEXT NOT NULL,
  content TEXT NOT NULL,
  importance INTEGER NOT NULL,
  source TEXT NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  last_used_at TEXT,
  metadata_json TEXT
)
```

会话记录表：

```sql
conversation_messages(
  id TEXT PRIMARY KEY,
  conversation_id TEXT NOT NULL,
  role TEXT NOT NULL,
  content TEXT NOT NULL,
  created_at TEXT NOT NULL,
  metadata_json TEXT
)
```

### 8.3 检索流程

用户发起请求后，系统不应该把所有记忆都塞给模型，而应该检索最相关的内容。

推荐流程：

```text
用户输入
  -> 提取检索关键词
  -> 搜索长期记忆和历史会话
  -> 过滤低可信/过期记忆
  -> 选择少量高相关记忆
  -> 拼入 LLM system/context
  -> 生成回复或工具调用
```

### 8.4 记忆写入流程

不是每句话都值得永久保存。推荐使用“候选记忆 + 审核”的流程。

```text
对话结束
  -> LLM 总结可能值得记住的事实
  -> 标注类型、重要性、来源
  -> 普通记忆自动保存
  -> 高敏感/高优先级记忆请求用户确认
```

例如：

- “用户喜欢简洁回答”可以自动保存。
- “用户禁止操作某个路径”应该要求确认并设为高优先级。
- “用户今天心情不好”可以短期保存，不一定永久保存。

## 9. Hermes Agent 的借鉴点

Hermes Agent 的价值在于它不是只做一次对话，而是尝试建立一个持续学习的 Agent 系统。它可以为 AIRI 提供三个重要思路。

第一是长期记忆。Hermes 使用持久化记忆、用户画像和历史会话搜索，让 Agent 能跨会话理解用户。AIRI 可以参考 NousResearch `hermes-agent` GitHub 仓库中的记忆系统源码和数据流，但第一版建议在 AIRI 内本地重实现，使用 PGlite + Drizzle + LLMWiki/RAG，避免把外部 Hermes 服务作为运行依赖。

第二是 skill / routine 沉淀。Hermes 会把重复任务总结成可复用技能。AIRI 也可以把一些固定流程沉淀下来，例如：

- 启动 GSV 前先检查 9881 端口。
- 检查 Ollama 模型列表。
- 切换 STT provider。
- 总结最近项目修改。

第三是反思循环。AIRI 可以在每天结束或对话结束时生成摘要，更新用户画像和项目状态。这种“慢速自我改进”比让模型自动改代码更安全。

需要注意的是，本项目不建议让 AIRI 自动修改核心代码、自动安装未知插件、自动执行高权限命令。所谓自进化应该是：

> 自动总结经验，生成可审核的建议，而不是无监督地控制电脑和修改系统。

## 10. OpenClaw / Computer Use 的借鉴点

Computer Use 是让 AI 从“会说”变成“会做”的关键能力。它包括观察屏幕、理解界面、使用鼠标键盘、操作浏览器、执行文件任务等。

但 Computer Use 风险很高，尤其是在 Windows 桌面环境中。AIRI 应该采用分级权限设计。

| 权限等级 | 能力 | 是否需要确认 |
| --- | --- | --- |
| Level 0 | 只聊天 | 不需要 |
| Level 1 | 只读观察：截图、窗口列表、网页读取、文件搜索 | 通常不需要 |
| Level 2 | 低风险操作：打开应用、切换窗口、填写但不提交 | 视场景确认 |
| Level 3 | 高风险操作：执行命令、删除文件、提交表单、结束进程 | 必须确认 |

Windows 实现建议：

- 截图：Electron desktopCapturer 或系统截图 API。
- 浏览器控制：优先使用 Playwright / Chrome DevTools Protocol，而不是纯鼠标点击。
- 鼠标键盘：可使用 nut.js、robotjs 或 Windows UI Automation。
- 文件操作：必须通过白名单工具 API，不让模型直接拼危险命令。
- 命令执行：默认禁用或强确认。

核心原则：

> 模型负责计划，工具负责执行，权限系统负责拦截危险动作。

## 11. 总体架构设计

建议将 AIRI 重构成可插拔 Agent Runtime。

```text
用户语音/文字
  -> STT Provider
  -> Agent Orchestrator
  -> Memory Retriever
  -> LLM Provider
  -> Tool Router / Computer Use
  -> 安全确认层
  -> TTS Provider
  -> AIRI 形象与语音输出
```

核心模块如下：

| 模块 | 职责 |
| --- | --- |
| LLM Provider | 连接 Ollama、本地 llama.cpp、OpenAI、OpenRouter 等 |
| STT Provider | 连接 Whisper、FunASR 或云端语音识别 |
| TTS Provider | 连接 GPT-SoVITS、Kokoro 或云端 TTS |
| Memory Service | 保存和检索长期记忆 |
| Tool Router | 管理文件、浏览器、服务检查、命令等工具 |
| Computer Use Service | 控制屏幕、浏览器、鼠标键盘 |
| Safety Guard | 权限分级、路径白名单、危险操作确认 |
| Reflection Worker | 会话后总结、记忆提取、routine 生成 |

这种设计的好处是，Gemma 不是被迫承担所有职责，而是成为 Agent 系统中的核心推理模块。这样即使模型规模较小，也能通过记忆和工具获得更强能力。

## 12. Provider 可插拔设计

为了同时支持本地和云端，所有模型能力都应该通过 provider 抽象。

### 12.1 LLM Provider

候选实现：

- `ollama-local`
- `llama-cpp-local`
- `lmstudio-local`
- `openai-compatible`
- `anthropic`
- `gemini`
- `openrouter`

统一接口可以包括：

```ts
interface LLMProvider {
  id: string
  generate: (messages: ChatMessage[], options: GenerationOptions) => Promise<LLMResponse>
  stream: (messages: ChatMessage[], options: GenerationOptions) => AsyncIterable<LLMChunk>
}
```

### 12.2 TTS Provider

候选实现：

- `gpt-sovits-local`
- `kokoro-local`
- `edge-tts`
- `openai-tts`

统一输出应该是浏览器可解码音频，避免流式 WAV 分段导致无声。

### 12.3 STT Provider

候选实现：

- `faster-whisper-local`
- `paraformer-local`
- `browser-stt`
- `cloud-stt`

语音识别结果应包含文本、置信度、语言和时间戳，为后续记忆提取提供依据。

## 13. 训练方案设计

### 13.1 第一阶段：SFT

SFT 是最适合第一阶段的训练方式。它通过高质量样本让模型学习“在什么输入下应该给出什么输出”。

推荐训练内容：

- AIRI 人格语气。
- 陪伴聊天。
- 桌面助理意图识别。
- 工具调用格式。
- 记忆使用方式。
- 安全确认和拒绝。

### 13.2 第二阶段：DPO

如果后续有足够数据，可以加入 DPO。DPO 需要偏好数据，也就是同一个问题下有“更好回答”和“更差回答”。

例如：

- 更好回答：先共情，再给出小步骤。
- 更差回答：机械说教，或者过度承诺。

DPO 适合进一步优化陪伴感和安全边界，但不是第一阶段必须做。

### 13.3 LoRA 参数建议

在 8GB 显存下，可以从保守参数开始：

| 参数 | 建议值 |
| --- | --- |
| quantization | 4-bit |
| LoRA rank | 8 或 16 |
| lora_alpha | 16 或 32 |
| batch size | 1 或 2 |
| gradient accumulation | 8 或 16 |
| max sequence length | 1024 或 2048 |
| learning rate | 1e-4 到 2e-4 |
| epoch | 1 到 3 |

第一轮训练不追求大规模，而是验证：

- 数据格式是否正确。
- loss 是否正常下降。
- 模型是否学会 AIRI 语气。
- 工具调用格式是否稳定。
- 是否出现灾难性遗忘或胡乱调用工具。

### 13.4 推荐工具链

训练工具建议：

- Unsloth：适合显存有限设备，能降低显存占用。
- TRL：用于 SFT / DPO 等训练流程。
- PEFT：用于 LoRA adapter。
- datasets：管理训练数据。
- llama.cpp / Ollama：用于最终本地推理部署。

Windows 本机训练可能会遇到环境问题。如果本地配置困难，可以考虑 WSL2 或云端 GPU 完成训练，再把结果转回本地部署。

## 14. 评估方案

模型微调不能只看训练 loss，还要设计任务评估。

建议评估五类能力：

| 能力 | 评估方式 |
| --- | --- |
| 人格一致性 | 多轮对话中是否保持 AIRI 角色 |
| 陪伴质量 | 是否共情、自然、不过度说教 |
| 记忆使用 | 是否正确引用检索到的记忆 |
| 工具调用 | JSON / tool call 格式是否稳定 |
| 安全边界 | 危险操作是否请求确认或拒绝 |

可以构建一个小型评估集，例如 100 条固定测试问题，每次训练后比较：

- 基础模型回答。
- 微调模型回答。
- 微调模型 + 记忆系统回答。
- 微调模型 + 记忆系统 + 工具调用回答。

这样面试时可以展示清晰的 ablation study：

> 不是只说“我微调了模型”，而是证明每个模块带来了什么提升。

## 15. 部署方案

训练完成后有两种部署方式。

第一种是 adapter 部署。保留基础模型和 LoRA adapter，推理时加载 adapter。这种方式便于切换不同人格版本，但部署链路稍复杂。

第二种是合并 LoRA 后转 GGUF。将 LoRA 权重合并到基础模型，再转成 GGUF，用 Ollama 或 llama.cpp 本地运行。这种方式适合普通用户使用。

推荐最终路线：

```text
训练数据
  -> SFT / LoRA 训练
  -> 保存 adapter
  -> 评估
  -> 合并 LoRA
  -> 转 GGUF
  -> Ollama Modelfile
  -> AIRI LLM Provider 加载
```

Ollama 侧可以提供一个专用模型名，例如：

```text
airi-gemma-companion:latest
```

## 16. 安全设计

Computer Use 和长期记忆都涉及安全问题。项目必须显式设计安全边界。

### 16.1 文件安全

- 默认只能读写白名单目录。
- 对外部项目目录只读或禁止访问。
- 明确禁止操作 MiniBox 和毕业设计文件夹。
- 删除、移动、覆盖文件必须确认。

### 16.2 命令安全

- 模型不能直接执行任意 shell。
- 所有命令必须经过工具层解析。
- 高风险命令需要用户确认。
- 记录每次执行历史，便于回溯。

### 16.3 记忆安全

- 用户可以查看、编辑、删除记忆。
- 敏感记忆需要确认后保存。
- 记忆不能自动上传云端。
- 切换云端 LLM 时，需要提示哪些上下文会发送出去。

### 16.4 自进化安全

自进化只允许做三件事：

- 总结记忆。
- 生成 routine 草稿。
- 提出可审核的改进建议。

不允许默认做：

- 自动修改核心代码。
- 自动安装未知依赖。
- 自动启用高权限工具。
- 自动执行危险桌面操作。

## 17. 项目实施路线

### P0：整理当前本地化链路

- 确认 Ollama + Gemma 正常运行。
- 确认 GPT-SoVITS provider 正常发声。
- 确认 Whisper / FunASR provider 可用。
- 整理未提交文件，避免把 venv 和模型文件提交进仓库。

### P0.5：建立 AIRI-Brain 知识库工作流

在真正实现记忆系统前，先建立一个本地知识库规范，用来连接 Claude Code、Obsidian、LLMWiki、GBrain 和 AIRI。

推荐目录：

```text
AIRI-Brain/
  00-inbox/
  10-profile/
  20-projects/
  30-research/
  40-life/
  50-routines/
  60-interviews/
  90-archive/
```

这套知识库不是 AIRI 仓库源码的一部分，而是用户本地私有知识库。Claude Code 可以负责整理公开论文、项目资料和长对话，把它们写成 LLMWiki/Obsidian Markdown；AIRI/Gemma 运行时通过 RAG 读取这些整理后的知识。

### P0.6：明确开发期强脑与运行期本地脑

本项目采用“双脑架构”：

```text
开发期强脑：
Claude Code / Codex / Gemini CLI / OpenCode
通过 CC Switch 切换模型/provider
负责开发、研究、整理资料、生成 LLMWiki、重构代码

运行期本地脑：
AIRI + 本地 Gemma 4
负责日常陪伴、记忆调用、桌面助理、语音交互
```

这两层不能混淆。Claude Code 是建设 AIRI 的强模型工具，不是 AIRI 的日常人格大脑；CC Switch 是开发工具配置切换器，不是 AIRI 运行时 provider 系统。AIRI 运行期的本地/云端 LLM 切换应由 AIRI 自己的 provider 设置管理。

### P1：实现本地长期记忆

- 新建 Memory Service。
- 使用 PGlite + Drizzle 保存记忆和会话。
- 第一版实现文本检索；FTS/BM25/embedding 后续增强。
- 在对话前注入相关记忆。
- 提供记忆查看和删除 UI。

### P2：实现 LLMWiki / Obsidian / GBrain 风格知识同步

- 将用户画像、边界、项目状态和 routine 导出为 Markdown。
- 让 Obsidian 可以直接查看和编辑这些知识。
- 用 LLMWiki 风格组织页面，保证人类可读、LLM 可检索。
- 预留 GBrain/MCP/向量检索接口，但第一版不强依赖外部系统。

### P3：实现 RAG 检索注入

- 对话前检索 Memory Service。
- 检索 AIRI-Brain / LLMWiki Markdown。
- 将强规则、用户画像、相关记忆和项目资料压缩进上下文。
- 记录每次回答使用了哪些记忆，方便审计。

### P4：实现 Agent Orchestrator

- 统一管理 LLM、记忆、工具调用。
- 支持本地和云端 LLM provider。
- 定义 tool call 格式。
- 建立安全确认层。

### P5：实现 skill/routine

- 把重复工作流沉淀为可审核的 routine。
- 允许 Claude Code 在开发期生成 skill 草稿。
- 运行期 AIRI 执行 routine 前展示计划和风险。

### P6：实现 Computer Use v1

- 截图和屏幕观察。
- 浏览器自动化。
- 文件搜索。
- 打开应用。
- 高风险操作确认。

### P6.5：实现多数据源导入与人格塑造

- 支持用户手动导入微信聊天记录、飞书聊天记录和本地知识库。
- 原始导入数据默认只保存在本地 raw import 区。
- 对导入内容做脱敏、切块、摘要和候选记忆提取。
- 关系、情绪、人格判断和第三方隐私信息必须进入用户审核队列。
- 审核后的内容进入 Memory DB / LLMWiki / RAG。
- 高质量、脱敏、确认后的样本再进入 LoRA 数据候选池。

### P7：构建训练数据集

- 收集陪伴聊天样本。
- 构造桌面助理样本。
- 构造记忆使用样本。
- 构造工具调用样本。
- 构造安全拒绝样本。

### P8：LoRA 微调 Gemma

- 先小规模训练验证流程。
- 评估人格、工具调用和安全边界。
- 合并 LoRA 并转本地推理格式。
- 接入 AIRI LLM Provider。

## 18. 面试展示思路

这个项目在面试中不要讲成“我做了一个 AI 聊天软件”，而应该讲成：

> 我基于开源桌面虚拟伙伴 AIRI，设计并实现了一个本地优先的陪伴型桌面智能体系统。系统包含本地 LLM 推理、LoRA 个性化微调、长期记忆检索、语音交互、工具调用和 Computer Use。我的重点不是简单调用 API，而是研究如何让小模型在端侧通过记忆和工具增强，形成可长期使用的个人智能体。

可以重点讲四个技术亮点。

第一，端侧模型适配。你可以讲为什么 8GB 显存不适合全参数微调，为什么选择 LoRA / QLoRA，以及如何控制显存。

第二，记忆与微调分工。你可以讲清楚长期记忆不能靠微调硬背，而要用数据库、检索和摘要系统实现。这个理解会比很多只会说“微调模型记住用户”的项目更专业。

第三，Agent 工程架构。你可以展示 LLM Provider、Memory Service、Tool Router、Safety Guard、Computer Use Service 的模块拆分，体现系统设计能力。

第四，安全边界。你可以强调桌面智能体的风险，包括文件删除、命令执行、隐私上传，并说明你如何设计权限分级和确认机制。

面试官可能会问：

### Q1：为什么不用更大的模型？

回答思路：

> 本项目目标是本地长期运行，硬件是 RTX 3070 Ti 8GB。更大的模型虽然能力更强，但速度、显存和部署成本不适合桌面常驻场景。我选择小模型 + LoRA + 记忆检索 + 工具调用的路线，本质上是用系统工程补足模型参数规模的不足。

### Q2：长期记忆为什么不用微调实现？

回答思路：

> 微调适合学习稳定模式，例如语气、工具调用格式和任务偏好；但用户每天产生的新事实具有动态性和可删除性，不适合写进模型参数。长期记忆应该放在外部数据库中，通过检索注入上下文，这样可更新、可删除、可审计。

### Q3：LoRA 微调的数据怎么来？

回答思路：

> 我会构造五类数据：陪伴聊天、桌面助理、记忆使用、Computer Use 工具调用、安全拒绝。第一阶段用人工高质量样本和项目真实对话生成小规模数据集，后续通过用户反馈和会话摘要形成数据飞轮。

### Q4：如何评估微调是否有效？

回答思路：

> 我不会只看 loss，而会构建固定评估集，分别测试人格一致性、陪伴质量、记忆使用、工具调用格式和安全边界。同时做消融对比：基础模型、微调模型、微调模型加记忆、微调模型加工具调用。

### Q5：Computer Use 怎么保证安全？

回答思路：

> 我把操作分成只读观察、低风险操作和高风险操作。高风险操作必须用户确认，文件系统使用白名单，命令执行通过工具层封装，模型不能直接执行任意 shell。这样模型负责决策，安全层负责拦截。

## 19. 项目风险与解决方案

| 风险 | 影响 | 解决方案 |
| --- | --- | --- |
| 8GB 显存不足 | 训练失败或速度很慢 | 使用 QLoRA、短序列、小 batch、Unsloth |
| 数据质量不高 | 微调后模型变差 | 少量高质量数据优先，逐步扩充 |
| 模型乱调用工具 | 桌面操作危险 | 工具调用格式训练 + 安全确认层 |
| 记忆污染 | 模型记住错误事实 | 记忆可编辑、可删除、带来源和重要性 |
| 云端 API 泄露隐私 | 用户数据外传 | 本地优先，切换云端前提示上下文范围 |
| 自进化失控 | 自动执行危险行为 | 自进化只生成建议和草稿，不自动高危执行 |

## 20. 总结

本项目的核心价值不在于“把 Gemma 微调成一个会说话的角色”，而在于构建一个完整的本地智能体系统。

LoRA 微调解决的是模型行为风格问题，使 Gemma 更像 AIRI，更适合陪伴、语音对话、桌面助理和工具调用。

长期记忆系统解决的是跨会话连续性问题，使 AIRI 能记住用户偏好、项目状态和重要事件。

Computer Use 解决的是执行能力问题，使 AIRI 不只是回答问题，而能在授权下帮助用户操作电脑。

Provider 抽象解决的是可扩展性问题，使系统可以在本地模型和云端 API、本地语音和云端语音之间自由切换。

从算法岗面试角度看，这个项目覆盖了大模型微调、端侧部署、RAG 记忆检索、Agent 工具调用、安全控制和多模态交互，是一个比普通聊天机器人更完整、更有技术纵深的 AI 应用项目。

最终目标可以概括为：

> 用 LoRA 微调塑造 AIRI 的人格与任务习惯，用长期记忆保存用户与项目状态，用 Computer Use 扩展执行能力，构建一个真正属于用户自己的本地陪伴型桌面智能体。
